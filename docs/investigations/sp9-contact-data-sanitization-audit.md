# SP-9: Contact Data Sanitization Audit

**Date:** 2026-02-22
**Scope:** All SolidJS frontend components in `ui/src/`
**Result:** PASS -- No XSS vulnerabilities found

## Summary

All contact field render paths in the vauchi desktop SolidJS frontend are
XSS-safe. SolidJS JSX text interpolation (`{value}`) escapes HTML entities
by default, and the codebase consistently uses this pattern for all
user-supplied data. The only `innerHTML` usage is for QR code SVG rendering,
which uses internally generated SVG from the Rust backend -- not user data.

## Methodology

1. Searched all `.tsx` and `.ts` files in `ui/src/` for unsafe DOM patterns:
   innerHTML, direct DOM manipulation, dynamic code execution, template
   literal HTML construction, and attribute injection vectors.
2. Reviewed every component that renders contact data (display_name, fields,
   labels, values, fingerprints).
3. Traced innerHTML data sources to their origin in the Rust backend.

## Findings

### 1. innerHTML Usage (4 occurrences) -- SAFE

All four occurrences are in `ui/src/pages/Devices.tsx`:

| Line | Expression | Source | Risk |
|------|-----------|--------|------|
| 415 | `innerHTML={state.qrSvg}` | `generate_device_link_qr` Tauri command | None -- SVG generated by Rust `generate_qr_svg()`, no user input in SVG |
| 472 | `innerHTML={state.qrSvg}` | Same as above (offline flow) | None -- same trusted source |
| 533 | `innerHTML={frames[...].svg}` | `generate_multipart_qr` Tauri command | None -- SVG generated by Rust `generate_qr_svg()` |
| 716 | `innerHTML={frames[...].svg}` | Same as above (join flow) | None -- same trusted source |

The `generate_qr_svg()` function in `src-tauri/src/commands/devices.rs`
(lines 543-582) constructs SVG purely from QR code module coordinates
(integer x/y positions). It does not interpolate any user-supplied strings
into the SVG markup. The input data is encoded into QR modules by the
`qrcode` crate, and only the resulting dark/light module grid positions are
written as `<rect>` elements.

All four usages have eslint-disable comments documenting the rationale:
`solid/no-innerhtml -- SVG is backend-generated, trusted`.

### 2. document.createElement Usage (1 occurrence) -- SAFE

`ui/src/pages/Settings.tsx` line 758-762:
```typescript
const a = document.createElement('a');
a.href = url;
a.download = 'vauchi-gdpr-export.json';
a.click();
```

This creates a download link for GDPR data export. The `url` is a
`URL.createObjectURL(blob)` from JSON data retrieved via the
`export_gdpr_data` Tauri command. No user data is injected into HTML. This
is a standard file-download pattern.

### 3. Contact Data Rendering -- ALL SAFE (JSX text interpolation)

Every component that displays user-supplied contact data uses SolidJS JSX
text interpolation, which auto-escapes HTML entities:

**Contacts.tsx** (primary contact display):
- `{contact.display_name}` (lines 431, 434, 495, 504, 519, etc.)
- `{field.label}` (lines 588, 594, 617, 627, 737)
- `{field.value}` (lines 588, 595)
- `{fingerprint()?.formatted_their}` (line 673)
- `{fingerprint()?.formatted_our}` (line 681)
- `{selectedContact()?.id.substring(0, 16)}` (line 644)
- Validation display text, trust level (lines 599-604)

**Home.tsx** (own card display):
- `{card()?.display_name}` (line 207)
- `{field.label}` (lines 231, 236, 245, 255, 265, 432)
- `{field.value}` (lines 231, 237)
- `{viewer.display_name}` (lines 381, 383, 388)
- `{identity()?.public_id.substring(0, 16)}` (lines 211-212)

**Exchange.tsx** (exchange flow):

**Recovery.tsx** (recovery flow):

**Help.tsx** (FAQ display):

**Setup.tsx** (initial setup):
- `{name()}` is only used as input value, never rendered as HTML

### 4. Other Patterns Checked -- SAFE

- **No React-style unsafe HTML injection**: Zero occurrences.
- **No template literal HTML construction**: Zero occurrences.
- **No `.textContent =` or `.innerText =`**: Zero occurrences.
- **No dynamic code execution**: Zero occurrences of `Function()` constructor or similar.
- **`src={qrImageUrl()}`** in Exchange.tsx line 193: Safe -- data URL generated by `qrcode` npm library from internal data.
- **`setTimeout` / `setInterval` calls**: All use arrow function callbacks with hardcoded behavior.

## Conclusion

The vauchi desktop frontend has **no XSS vulnerabilities** in contact data
rendering paths. SolidJS's default behavior of escaping JSX text
interpolation provides a strong baseline defense, and the codebase does not
circumvent this protection for any user-supplied data. The only `innerHTML`
usage is for QR SVG rendering from trusted Rust backend output, which is
appropriately documented with eslint-disable comments.

No remediation actions are required.
